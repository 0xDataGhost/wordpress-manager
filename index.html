01 — The problem

Running a store from wp-admin creates real friction.

The default WooCommerce admin mixes e-commerce work with WordPress internals — slow, cluttered, and unsafe to delegate.

🔓

No separation of duties

Anyone who touches orders effectively gets the keys to the entire site.

🙅

No safe delegation

You can't grant "refund + reply to reviews" without exposing plugins, users, settings.

🕳️

No visibility

No unified audit of who changed what, no clean fulfillment view, no multi-store overview.

🔑

One store, one login

Several stores means several wp-admin logins and no shared team model.

02 — The solution

A dedicated operations dashboard beside WooCommerce.

A fast, purpose-built, Arabic-first (RTL) interface for exactly the e-commerce tasks your team does every day — with real roles and a full audit trail — while WordPress itself stays locked down.

🏢

Multi-tenant

One deployment serves many independent stores — each with its own team, permissions, data, and WooCommerce connection, fully isolated.

🎛️

Purpose-built

Only the daily e-commerce surfaces, on a UI designed for the job — not the whole sprawling WordPress admin.

🛡️

Locked-down WordPress

A thin connector plugin executes commands; all logic lives in the SaaS and payment secrets never leave the store.

06 — Architecture

One safe pipeline to WordPress.

The dashboard never edits WordPress directly. Every change flows through a command outbox — the heart of the system.

┌─────────────────────┐      signed, idempotent commands      ┌──────────────────────┐
│   React Dashboard    │  ─────────  (HMAC + keys)  ─────────▶ │  WordPress Connector  │
│   Arabic RTL SPA     │                                       │   thin WP plugin      │
│          │           │  ◀─────────  real-time webhooks  ──── │          │            │
└──────────┼───────────┘                                       └──────────┼───────────┘
           │                                                              │
   ┌───────▼────────────┐         mirror + command outbox        ┌────────▼───────────┐
   │   SaaS API (Node)   │  ───────────────────────────────────  │  WooCommerce store  │
   │   Postgres · Redis  │                                        │  (source of truth)  │
   └─────────────────────┘                                        └─────────────────────┘
STEP 1

Recorded first

Every change is written to the outbox before it's attempted — nothing is lost if WordPress is briefly down; failures retry from the Command Center.

STEP 2

Executed safely

Sent to the connector signed and idempotent, so a retry can never apply a change twice — this is what makes money refunds safe.

STEP 3

Kept in sync

Webhooks mirror WordPress back in real time; reconciliation re-pulls anything a lost webhook missed.

i
Idempotent

A retry — even after a timeout — never applies a change twice.

e
Echo-suppressed

A change's own webhook isn't re-processed as an external edit. No loops.

c
Compare-and-set

A concurrent wp-admin edit yields a "refresh & retry" conflict, never a silent overwrite.

h
Self-healing

Reconciliation detects drift vs. WooCommerce and re-pulls the affected data.

07 — Security & trust

Audited adversarially before release.

Security is the center of the design — not an afterthought.

✓ Tenant isolation

No cross-store access

Every record is scoped to its store; audited clean. No store can ever read or touch another's data.

✓ Payment secrets

Never leave WordPress

Enable/disable gateways from the dashboard — but API keys & secrets are stripped at the connector and again at the SaaS.

✓ Money-safe refunds

No double-spend

A stable idempotency key end-to-end plus a database in-flight guard make double-refunds impossible, even under retries.

✓ Signed transport

HMAC + replay window

Every request between the SaaS and WordPress is signed and time-boxed against replay.

✓ Full audit trail

Who / what / when

Every sensitive action is logged — ids and counts only, never secrets or PII values.

✓ Least privilege

Per-action permissions

Money and config actions sit behind their own stricter, separately-grantable permissions.
